Object Oriented Programming
OOP stands in contrast with procedural or functional programming, though the styles can intermix. The fundamental notion of OOP is that there are many advantages to be had by bundling functions together with the data they act on. Instead of dumb structs, we have intelligent, self contained objects. This encapsulation allows objects to be reasoned about in isolation, and allows clients of objects to view the object as a black box. This encapsulation leads to a program that is easier to think about, and also leads to an easy algorithm for how to break down a complex project: simply determine the objects that the overall project is build out of, create each object, then wire them together.
Object
A bundle of data and methods to act on that data.
Class
A blueprint specifying how to create objects of a particular type.
Instance
An object build from a particular class
Method
A function that is bundled into an object.
Module
This means different things to different people, but generally it means some component of a larger system that has has a well defined interface such that it can be reused across project.
Library
A library is a non-volatile set of resources that provide some functionality through a well defined interface in such a way that it can be easily integrated into new projects.
Class Variables
As opposed to an instance variable, only a single copy of a class variable exists, no matter how many instances of the class have been created. Also known as static member variables.
Inheritance
Inheritance is one method for extending the functionality of a class. When making a subclass, one guarantees that the subclass will behave in a way compatible with the superclass. Unless you need to do some polymorphic weirdness, it's almost always inferior to simple composition.
Method overriding
Subclasses can add new methods, but they can also override methods supplied by a superclass.
